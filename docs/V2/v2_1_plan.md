# Hyself V2.1 规划文档

## 1. 文档目标

本文件用于定义 Hyself 在 V2 发布完成后的下一阶段工作范围。

V2 已经完成结构升级、自动化验证、浏览器回归与人工验收，V2.1 不再以“大规模重写”为目标，而是围绕以下三个方向推进：

1. 收口 V2 中保留的兼容层和过渡结构
2. 在现有稳定边界上补齐真正有价值的能力增强
3. 把当前可重复执行的本地回归能力提升为持续化工程基线

## 2. 阶段定位

V2.1 的角色不是新平台重构，而是“稳定化 + 强化版”。

如果说 V2 解决的是“系统能否从 V1 升级到可持续演进结构”，那么 V2.1 要解决的是：

1. 现有结构是否已经足够清晰，能支持后续持续开发
2. 聊天、资源中心、权限、娱乐中心是否已经具备继续扩展的稳定接口
3. 浏览器回归、测试、发布说明、规划文档是否已经形成长期维护闭环

## 3. 当前基线

截至 V2 发布完成，当前已具备以下基础：

1. 前端聊天中心已拆成场景 composable 与多份 store/runtime 模块
2. 后端 chat 已物理落地 interfaces、application、domain、infrastructure 结构
3. 资源中心、头像、聊天附件已开始共享统一资产引用能力
4. WebSocket 领域广播已统一围绕 event envelope 收口
5. RBAC、错误页、娱乐中心落地、聊天主链路已完成自动化回归和人工验收

这意味着 V2.1 可以避免重复做结构性大手术，而应该优先做“清理旧壳 + 补关键能力 + 固化工程基线”。

## 4. V2.1 核心目标

V2.1 目标分为三组：

### 4.1 结构收口

1. 清理 V2 为兼容保留的顶层导出壳和过渡模块
2. 继续减少超大视图 / 超大接口文件中的查询编排逻辑
3. 让现有文档与代码状态重新一致

### 4.2 能力增强

1. 强化前端实时事件统一消费层
2. 增强聊天消息类型与资产接入能力
3. 推进聊天搜索、审核、媒体能力的下一阶段落地

### 4.3 工程基线

1. 把当前浏览器回归能力继续工程化，减少仅靠本地手跑的依赖
2. 持续补齐测试和发布说明，使版本演进可以复用
3. 为 V2.2 或更长期方向建立更稳定的交付节奏

## 5. 工作范围

### 5.1 P0 必做

#### A. 文档与治理同步

目标：让文档重新准确反映当前代码现状。

范围：

1. 更新 [v2_architecture.md](v2_architecture.md) 中已落地但仍写成“目标态”的部分
2. 更新 [chat_v2_refactor_plan.md](chat_v2_refactor_plan.md) 中已过时的“仍待完成”描述
3. 统一 V2、V2.1、发布说明、收尾清单之间的术语与边界定义

验收标准：

1. V2 文档不再出现与当前代码结构明显冲突的描述
2. 新成员只看文档即可理解 V2 已完成项与 V2.1 目标项

#### B. 资源中心继续瘦身

目标：让资源中心接口进一步靠近 command/query + payload service 结构。

范围：

1. 继续拆分 [hyself/views.py](../../hyself/views.py) 中剩余查询编排
2. 把列表、搜索、回收站、虚拟目录相关读取逻辑继续下沉到 query/service
3. 保持现有接口路径和响应格式兼容

验收标准：

1. `hyself/views.py` 的职责进一步收缩到请求协调与响应返回
2. 不引入新的权限回退、回收站回归或资源列表展示回归

#### C. Chat 顶层兼容层处理

目标：明确顶层兼容壳是否继续保留，以及保留多久。

范围：

1. 评估 [chat/views.py](../../chat/views.py) 与 [chat/serializers.py](../../chat/serializers.py) 是否仍有外部依赖
2. 如无必要，直接移除兼容壳
3. 如暂时保留，显式标记为过渡层并写入文档

验收标准：

1. chat 接口层不再存在“长期过渡但无人维护”的壳文件
2. 兼容策略在文档中有明确说明

### 5.2 P1 应做

#### D. 前端 Realtime 统一消费层

目标：进一步降低页面和 store 直接消费 WS 事件的分散程度。

范围：

1. 继续收口 websocket event decoder / dispatcher
2. 让聊天、权限、系统通知等实时事件都优先通过统一消费层进入 store
3. 减少页面层与业务组件对原始 WS 事件结构的感知

验收标准：

1. 新增聊天事件不需要在多个页面/组件中重复兼容处理
2. 实时回归测试继续通过

#### E. 消息能力增强

目标：在已接通资产能力的前提下，补齐下一阶段消息能力。

范围：

1. 增强图片消息、文件消息及相关媒体预览链路
2. 统一 chat asset picker、附件选择和媒体展示接口
3. 为后续更完整的消息能力演进保留稳定数据结构

验收标准：

1. 图片/文件消息链路不再依赖散落兼容逻辑
2. 资源中心和聊天对资产引用的复用更清晰

#### F. 搜索与审核增强

目标：把 V2 里已完成骨架的搜索与审核能力再往前推一步。

范围：

1. 强化聊天搜索适配层与查询边界
2. 继续细化 moderation 相关规则与展示边界
3. 为后续内容治理预留更稳定的扩展点

验收标准：

1. 搜索与审核链路的规则不再散落在接口和页面中
2. 回归范围覆盖搜索与审核关键路径

### 5.3 P2 可选

#### G. 资产域独立演进预研

目标：为资源中心从 `hyself` 继续演进为独立资产域做准备。

范围：

1. 梳理 `Asset` / `AssetReference` 的接口边界
2. 评估是否把部分资源中心能力抽成独立 `assets` 域
3. 为对象存储或更复杂媒体处理保留扩展接口

#### H. 浏览器回归接入 CI

目标：把当前本地可跑的浏览器回归进一步沉淀为持续集成基线。

范围：

1. 明确 smoke data 准备与清理策略
2. 在 CI 中接入聊天、RBAC、错误页/娱乐中心三类浏览器回归
3. 统一失败日志、截图、trace 的保留方式

## 6. 不纳入 V2.1 的事项

以下内容建议继续延后，不纳入 V2.1 主范围：

1. 音视频通话
2. 真正的全文检索引擎接入
3. 完整的单条消息级已读回执闭环
4. 多端同步冲突解决算法
5. 大规模对象存储切换实施

这些内容要么风险高，要么需要新的基础设施前提，不适合作为 V2.1 的主要交付目标。

## 7. 建议里程碑

### M1 文档与结构收口

目标：文档、兼容层和资源中心视图结构完成第一轮收口。

验收标准：

1. V2 系列文档已同步现状
2. chat 顶层兼容层策略明确
3. 资源中心视图继续瘦身且关键回归通过

### M2 Realtime 与消息增强

目标：统一 realtime 消费层，并把消息能力增强推进到可验收状态。

验收标准：

1. 实时事件消费边界更单一
2. 图片/文件消息链路增强完成
3. 浏览器与后端回归覆盖新增路径

### M3 搜索/审核与工程固化

目标：完成搜索/审核增强，并把浏览器回归进一步工程化。

验收标准：

1. 搜索与审核边界更清晰
2. 浏览器回归可稳定复用
3. V2.1 发布说明与验收清单形成闭环

## 8. 发布门槛

V2.1 若要发布，建议至少满足以下条件：

1. 前端 type-check、Vitest 通过
2. 后端 chat/user/hyself/game 关键回归通过
3. 浏览器聊天、RBAC、错误页/娱乐中心回归通过
4. 相关文档与发布说明同步完成

## 9. 总结

V2.1 的核心不是再做一次 V2 级别的架构搬迁，而是在已经通过验收的 V2 基线上，继续把系统变得更清晰、更稳定、更适合继续迭代。

判断 V2.1 是否成功，不取决于做了多少新功能，而取决于三件事：

1. 过渡结构是否被进一步清理
2. 新能力是否建立在稳定边界之上
3. 工程验证是否比 V2 阶段更容易重复执行
